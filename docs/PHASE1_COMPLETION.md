# Phase 1 completion report

Prepared for Athena Huo at the close of Phase 1.

## Result

Phase 1 is engineering complete and both exit benchmarks are met. On the 62
question evaluation set across both data sources, factual accuracy is 95.2 percent
and traceability is 100 percent, every number in every answer traces to a real
query result. Average cost per query is 0.0083 dollars and p95 cost is 0.0092
dollars, both well under the 0.05 dollar gate. P95 model latency is 5.2 seconds,
under the 6 second target, with an average of 3.4 seconds. The gate passes,
including the cost condition.

## What shipped across Phase 1

- **Win probability.** Elo into Dixon-Coles into Monte Carlo, materialized into
  `team_ratings` and `match_predictions`, answered by the grounded pipeline from
  stored rows. A walk forward backtest reports honest accuracy, which is near
  uniform on the single tournament of data, with a clear caveat. No probability is
  generated by a language model.
- **Cost controls.** Token usage and cost are measured per query and surfaced in
  the eval report, with cost now a hard gate condition. A semantic answer cache
  replays previous answers, and per IP rate limiting caps abuse. Both use a separate
  trusted parameterized write path, never the model SQL path.
- **Authentication.** Supabase Auth with email and password, server and browser
  clients, and session refresh. Signed in users are rate limited per user id with a
  higher allowance; anonymous traffic per IP. The product stays usable signed out.
- **Personalization.** Signed in users follow teams and players, with grounded
  suggested questions built only from entity names. The user id always comes from
  the server session, never from request input.
- **Multilingual output.** Answers are written in the chosen language while every
  number stays grounded and traceable. The cache is language aware.
- **API-Football as a labelled second source.** The provider maps the licensed
  commercial feed into the neutral types. Both feeds live in one database, separated
  by a `source` column and an `af:` id prefix. The model knows which feed carries
  shot detail and the derived metrics and which does not, and returns a truthful no
  data answer when a metric is unavailable.
- **Performance hardening.** Precomputed per player and per team metric totals let
  leaderboards answer from a small table within the executor timeout. A previously
  slow leaderboard now returns in well under a second and matches the live join.
- **Housekeeping.** Middleware renamed to proxy for Next.js 16, a migration tracking
  runbook, and prediction answers carry the entertainment framing in the UI.

## The embedder deployment finding

The query time embedder runs a local transformers model in process. Measured cold
load plus first embed is about 0.7 seconds with the model already on disk, and a
warm embed is 3 milliseconds. On Vercel serverless this is a real risk: a fresh
container has no model on disk, so the first cold request also downloads the model
from the Hugging Face CDN, which adds a few seconds, and the heavy
`@xenova/transformers` plus `onnxruntime-web` dependency inflates the function
bundle. On cold starts this can push p95 latency up and, on a 10 second function
limit, risk a timeout.

Recommendation, not implemented in this pass because it needs your sign off on the
approach: move query time embedding off the in process local model, either to a
small hosted embeddings call or to a dedicated always warm embedding service, for
example on Railway which is already in the stack. Either option removes the cold
start model load from the request path and also removes the `@xenova/transformers`
dependency, which carries the critical `protobufjs` advisory below. The replacement
must use the same model, `all-MiniLM-L6-v2` at 384 dimensions, so the stored
glossary and cache vectors stay compatible. Warm steady state is already fast, so
this is specifically a cold start and dependency hygiene improvement, not a
correctness fix.

## Security advisories

`npm audit` reports six advisories, none with a non breaking fix. They require
breaking downgrades, so none were applied:

- `postcss` below 8.5.10, moderate, a CSS stringify XSS. The only offered fix
  downgrades Next.js to version 9. Low real risk here since the app has no user
  controlled CSS. Wait for a Next.js patch that bumps postcss.
- `protobufjs`, critical and high, reached through
  `@xenova/transformers` to `onnxruntime-web` to `onnx-proto`. The only offered fix
  downgrades transformers. Low real exploitability since we run a known local model,
  not untrusted protobuf input. Moving the embedder to a hosted service, as above,
  removes this dependency entirely.
